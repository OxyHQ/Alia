# Model abstraction

Alia exposes one stable namespace of model identifiers and routes them internally to whichever provider model is cheapest, fastest, and healthy. Clients, errors, and analytics never see the underlying provider. This is a shipping blocker — violating it is a security and a brand bug at once.

## Public Alia models

Defined in `apps/api/src/internal/providers/lib/alia-models.ts` as `ALIA_MODELS`. Every model carries a tier, category, credit multiplier, and capability flags.

| Tier | ID | Category | Notes |
|------|-----|----------|-------|
| `lite` | `alia-lite` | general | Fast, low-cost; `0.5×` credit multiplier |
| `v1` | `alia-v1` | general | Default balanced tier; `1×` |
| `v1-pro` | `alia-v1-pro` | general | Higher quality general tier |
| `v1-pro-max` | `alia-v1-pro-max` | general | Top-tier general capability |
| `v1-codea` | `alia-v1-codea` | coding | Coding-tuned (Codea + CLI) |
| `v1-cowork` | `alia-v1-cowork` | general | Cowork desktop assistant tier |
| `v1-browser` | `alia-v1-browser` | general | Headless-browser-aware tier |
| `v1-vision` | `alia-v1-vision` | vision | Image understanding |
| `v1-audio` | `alia-v1-audio` | audio | Audio transcription / analysis |
| `v1-tts` | `alia-v1-tts` | audio | Text-to-speech |
| `v1-image` | `alia-v1-image` | multimodal | Image generation |
| `v1-multimodal` | `alia-v1-multimodal` | multimodal | Mixed audio / vision / text |
| `v1-voice` | `alia-v1-voice` | voice | Real-time voice |
| `v1-voice-pro` | `alia-v1-voice-pro` | voice | Pro real-time voice |

The `chatVisible` flag controls whether a model appears in the picker; `creditMultiplier` is how usage is billed.

## Public API

`GET /v1/models` returns only Alia identifiers:

```json
{
  "object": "list",
  "data": [
    {
      "id": "alia-v1",
      "object": "model",
      "owned_by": "alia",
      "name": "Alia V1",
      "category": "general",
      "is_default": true,
      "is_available": true,
      "capabilities": { "tools": true, "vision": true, "max_tokens": 8192 },
      "pricing": { "credit_multiplier": 1 }
    }
  ]
}
```

Query params:

- `category` — `general | coding | vision | audio | multimodal | voice`
- `chat=true` — return chat-visible models only

`GET /v1/models/:modelId` returns a single descriptor with the same shape. Provider model IDs, owners, and routing data are never included.

## Internal mapping

Each `AliaModel` is backed by a list of `ModelMapping` entries:

```ts
interface ModelMapping {
  provider: string;
  modelId: string;
  priority: number;
  qualityScore: number;
  pricingTier: 'free' | 'freemium' | 'paid';
  costPer1MInput?: number;
  costPer1MOutput?: number;
  costPerMinute?: number;
  averageLatencyMs?: number;
  capabilities: ModelCapabilities;
}
```

Mappings are generated by `apps/api/src/internal/providers/lib/generate-model-mappings.ts` and live in the internal `providers/` directory — never imported from outside `apps/api/src/internal/`.

## Fallback engine

`apps/api/src/internal/providers/lib/fallback-engine.ts` selects a provider per request:

1. Filter mappings by required capabilities (tools, vision, voice, context length)
2. Sort by `pricingTier` (free first), then `priority`, then `qualityScore`
3. Drop providers marked unhealthy by `provider-health.ts`
4. Try the top candidate; on 5xx, rate-limit, or circuit-breaker trip, escalate
5. Emit `alia.model_switch` if the provider changes mid-stream
6. After the run, update `averageLatencyMs` and the health state for that provider

The user sees a single response and a single Alia model name. Internal switches are invisible except for the optional `alia.model_switch` event.

## Sanitization

`sanitizeMessage()` in `apps/api/src/lib/errors/sanitize.ts` is the only sanctioned exit point for error strings reaching users. It strips:

- Provider names registered in `apps/api/src/internal/providers/lib/provider-names.ts`
- Provider-specific model IDs
- Provider error codes that hint at the upstream
- API endpoint paths from `*.openai.com`, `*.anthropic.com`, `*.googleapis.com`, etc.

Use it everywhere a thrown error might be serialized into an HTTP response, an SSE event, a notification, or a log line that ships to a customer dashboard.

```ts
import { sanitizeMessage } from '@/lib/errors/sanitize';

try {
  await someAIProviderCall();
} catch (err) {
  throw new ApiError(sanitizeMessage(err), { code: 'model_unavailable' });
}
```

## Rules of engagement

1. **Never** import from `apps/api/src/internal/providers/*` outside `apps/api`
2. **Never** name a provider in UI, API response, error message, log line, doc, or analytics event
3. **Never** add an endpoint that resolves an Alia ID to a provider model ID
4. **Never** ship a model picker that shows provider attribution
5. **Always** look up display strings with the public `ALIA_MODELS` registry, not the mapping table
6. **Always** route user-facing errors through `sanitizeMessage()`

Removed endpoints — there is no compatibility layer:

- `POST /v1/resolve-model` → `410 Gone`
- `POST /codea/resolve-model` → `410 Gone`
- `POST /v1/report-usage` → `410 Gone`
- `POST /codea/report-usage` → `410 Gone`

## Adding a new Alia model

1. Add the `AliaModel` entry to `ALIA_MODELS` in `apps/api/src/internal/providers/lib/alia-models.ts`
2. Add provider mappings in `generate-model-mappings.ts`
3. Update `model-capabilities-data.ts` if the model exposes a new capability
4. Add an entry to the model-store enum in `apps/app/lib/stores/model-store.ts`
5. Wire the picker label and emoji in the public list
6. Add a test in `apps/api/src/internal/providers/lib/__tests__/`

Never expose the new provider model ID, latency data, or cost numbers in a public response.
